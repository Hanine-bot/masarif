# تقرير المشروع

## 1. عنوان المشروع

LedgerLite: نظام مبسط لإدارة المعاملات المحاسبية

## 2. مقدمة

هذا المشروع عبارة عن تطبيق ويب صغير تم تطويره في مجال المحاسبة بهدف رقمنة عملية تسجيل الإيرادات والمصاريف اليومية بطريقة بسيطة واحترافية. الفكرة الأساسية هي الانتقال من التسجيل اليدوي إلى نظام رقمي يسمح بإدخال البيانات، حفظها، تعديلها، حذفها، ثم استخراجها في ملفات قابلة للاستعمال لاحقًا.

المشروع يحقق مفهوم **Vertical Slice** لأننا نمر بمسار كامل يبدأ من واجهة المستخدم، ثم منطق المعالجة داخل Python، ثم تخزين البيانات في قاعدة البيانات، ثم عرض النتائج من جديد داخل التطبيق.

## 3. هدف المشروع

يهدف المشروع إلى حل مشكلة شائعة في العمل المحاسبي اليومي، وهي صعوبة متابعة المعاملات المالية عندما تكون مسجلة يدويًا أو بشكل عشوائي. لذلك يوفر التطبيق وسيلة منظمة من أجل:

- تسجيل الإيرادات والمصاريف
- حساب الرصيد الصافي
- البحث عن المعاملات وتصفيتها
- حفظ البيانات في قاعدة بيانات
- استخراج نسخ احتياطية بصيغ متعددة

## 4. الأدوات والتقنيات المستعملة

- لغة البرمجة: Python
- إطار العمل: Flask
- قاعدة البيانات: SQLite
- تنسيقات الملفات: CSV و JSON و TXT
- التصميم: HTML و CSS
- البرمجة الكائنية: OOP

تم اختيار SQLite لأنها قاعدة بيانات خفيفة وسهلة الدمج داخل مشروع أكاديمي صغير، كما أنها مناسبة جدًا لإنجاز مشروع بسيط لكن منظم واحترافي.

## 5. فكرة المشروع المحاسبية

التطبيق موجه لتسيير المعاملات المحاسبية اليومية لمكتب صغير أو نشاط تجاري بسيط. كل معاملة تحتوي على:

- تاريخ العملية
- نوع العملية: دخل أو مصروف
- الصنف
- الوصف
- المبلغ
- طريقة الدفع
- مرجع العملية
- ملاحظات إضافية

وبعد تسجيل هذه البيانات يستطيع المستخدم متابعة الوضع المالي العام من خلال لوحة تحكم تعرض مجموع الإيرادات، مجموع المصاريف، وعدد العمليات، إضافة إلى الرصيد النهائي.

## 6. خطوات إنجاز المشروع

### الخطوة 1: تحليل المطلوب

في البداية تم تحليل ملف المشروع المرفق واستخراج الشروط الأساسية، وهي:

- استعمال Python
- توفير معالجة للملفات
- استعمال قاعدة بيانات
- إنشاء واجهة ويب بـ Flask
- تطبيق OOP

وبعد ذلك تم اختيار مجال **المحاسبة** لأنه ينسجم مع التخصص ويوفر فكرة عملية واضحة وقابلة للتنفيذ بسهولة.

### الخطوة 2: تحديد موضوع بسيط واحترافي

تم اختيار موضوع **إدارة المعاملات المحاسبية** لأنه:

- بسيط من حيث الفكرة
- واضح من حيث المدخلات والمخرجات
- مناسب جدًا للعرض الأكاديمي
- يسمح بتطبيق CRUD وقاعدة البيانات والتقارير بسهولة

### الخطوة 3: بناء الهيكل العام للمشروع

تم تقسيم المشروع إلى ملفات منظمة حتى يكون واضحًا وسهل الفهم:

- `app.py` لتشغيل التطبيق
- `routes.py` لتسيير الصفحات والتنقل
- `services.py` لتطبيق منطق المعالجة
- `repositories.py` للتعامل مع قاعدة البيانات
- `models.py` لتعريف الكيان الأساسي وهو المعاملة
- مجلد `templates` للواجهات
- مجلد `static` للتنسيق

هذا التقسيم يعطي المشروع شكلًا احترافيًا بدل وضع كل شيء داخل ملف واحد.

### الخطوة 4: إنشاء النموذج الكائني Transaction

تم إنشاء class باسم `Transaction` لتمثيل المعاملة المحاسبية. هذا الكلاس يحتوي على خصائص المعاملة ودوال التحقق من صحة البيانات وتحويلها إلى سجل قابل للحفظ أو العرض.

بهذا الشكل أصبح لكل معاملة تمثيل منظم داخل المشروع بدل التعامل مع البيانات بطريقة عشوائية.

### الخطوة 5: إنشاء طبقة قاعدة البيانات

تم إنشاء class باسم `TransactionRepository` وهو المسؤول عن جميع العمليات المرتبطة بقاعدة البيانات، مثل:

- إنشاء الجدول
- عرض المعاملات
- جلب معاملة محددة
- إضافة معاملة
- تعديل معاملة
- حذف معاملة
- حساب الملخص المالي

هذه الطبقة تفصل قاعدة البيانات عن باقي أجزاء المشروع، وهو أمر مهم من الناحية الهندسية.

### الخطوة 6: إنشاء طبقة الخدمات

تم إنشاء class باسم `AccountingService` ليكون المسؤول عن منطق العمل الفعلي. في هذه الطبقة يتم:

- بناء المعاملة انطلاقًا من بيانات الفورم
- التحقق من صحتها
- إرسالها إلى repository
- تصدير البيانات إلى ملفات
- استيراد النسخ الاحتياطية

هذا يعني أن service يمثل الوسيط بين واجهة المستخدم وقاعدة البيانات.

### الخطوة 7: إنشاء واجهة Flask

تم بناء واجهة ويب باستخدام Flask تحتوي على:

- صفحة رئيسية تعرض الملخص المالي وقائمة المعاملات
- صفحة لإضافة معاملة جديدة
- صفحة لتعديل معاملة
- صفحة لإدارة النسخ الاحتياطية

الواجهة صممت بشكل بسيط لكن منظم حتى يظهر المشروع بمظهر احترافي دون تعقيد غير ضروري.

### الخطوة 8: إضافة إدارة الملفات

لتحقيق جزء معالجة الملفات من المطلوب، تم إضافة إمكانيات:

- تصدير البيانات إلى CSV
- تصدير البيانات إلى JSON
- إنشاء تقرير TXT
- استيراد نسخة احتياطية من CSV أو JSON

وهذه الوظائف مفيدة في المحاسبة لأنها تسمح بحفظ البيانات واسترجاعها وإعداد تقارير بسيطة.

### الخطوة 9: إعداد التوثيق

تم إعداد ملفات توثيق لشرح المشروع، وهي:

- تقرير تقني بالإنجليزية
- هذا التقرير بالعربية
- README لتوضيح طريقة تشغيل المشروع

كما تم إضافة مخطط UML يوضح العلاقة بين الطبقات الأساسية.

## 7. كيف يعمل المشروع خطوة بخطوة

عند استعمال التطبيق يحدث التسلسل التالي:

1. يدخل المستخدم بيانات المعاملة من خلال الواجهة.
2. تنتقل البيانات إلى `routes.py`.
3. يقوم `AccountingService` ببناء كائن `Transaction`.
4. يتم التحقق من صحة البيانات.
5. إذا كانت البيانات صحيحة، يرسلها `AccountingService` إلى `TransactionRepository`.
6. يقوم `TransactionRepository` بحفظها داخل SQLite.
7. عند العودة إلى الصفحة الرئيسية، يتم جلب كل المعاملات وحساب الملخص المالي.
8. تعرض النتائج داخل لوحة التحكم.

وعند التصدير أو الاستيراد، يمر المشروع بنفس المنطق مع إضافة معالجة الملفات حسب الصيغة المختارة.

## 8. شرح الـ Diagram

الـdiagram جيد جدًا من حيث الفكرة والتنظيم، لأنه يوضح الطبقات الرئيسية الثلاث:

- `AccountingService`
- `TransactionRepository`
- `Transaction`

وهذا التقسيم منطقي وصحيح لأن:

- `Transaction` يمثل البيانات
- `TransactionRepository` يتعامل مع قاعدة البيانات
- `AccountingService` يحتوي منطق المشروع

### رأيي فيه

الـdiagram منظم وواضح ويعطي انطباعًا احترافيًا، خاصة أنه يبين اتجاه العلاقة من الطبقة العليا إلى الطبقة السفلى:

- الخدمة تعتمد على repository
- repository يعتمد على transaction

### ملاحظات بسيطة لتحسينه

- من الأفضل ذكر أن هذا **Class Diagram خاص بالباك-إند** فقط، لأنه لا يبين صفحات Flask أو الواجهة.
- إذا أردتِ نسخة أقوى أكاديميًا، يمكن إضافة `Routes / Flask Layer` في Diagram آخر منفصل.
- يمكن أيضًا إضافة نوع القيمة المرجعة لبعض الدوال، لكن هذا اختياري وليس ضروريًا.

بصفة عامة: **الـdiagram ناجح ومناسب جدًا للتقرير**.
